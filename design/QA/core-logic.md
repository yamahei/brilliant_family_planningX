品質管理：コアロジック
======================

基本的な観点
------------

### ONCE_YEARMONTHS(単年月:特定の年月に1回だけ発生する)

- 特定の年月だけで発生する

### EVERY_MONTH(毎月：毎月発生する)

- 毎年、毎月発生する

### SOME_MONTHS(毎年[m]月：毎年特定の月（複数可）に発生する)

- 毎年、指定月（月指定：任意の1つ）に発生する
- 毎年、指定月（月指定：1月）に発生する
- 毎年、指定月（月指定：12月）に発生する
- 毎年、指定月（月指定：連続する2つ）に発生する
- 毎年、指定月（月指定：連続する3つ）に発生する
- 毎年、指定月（月指定：連続する4つ）に発生する
- 毎年、指定月（月指定：連続しない2つ）に発生する
- 毎年、指定月（月指定：連続しない3つ）に発生する
- 毎年、指定月（月指定：連続しない4つ）に発生する

### SOME_YEARMONTHS(y年ごと毎年[m]月：y年ごと特定の月（複数可）に発生する)

- 開始年（StartYear）と一致する年の指定月に発生する
- 開始年 + Step（周期）の年の指定月に発生する
- 開始年 + Stepの間の年（発生しないはずの年）には発生しない
- 指定された複数の月（例：6月と12月）両方で正しく発生する

反転（NOT条件）の観点
-------------------
※全ルール共通

- 各ルールの `Not: true` 時、本来発生するはずの年月で「発生しない」こと
- 各ルールの `Not: true` 時、本来発生しないはずの年月で「発生する」こと
- **【重要】** `Not: true` の反転効果は、あくまで有効な期間（From-To）の内側だけで適用される。From-Toの期間外の年月は、Notフラグに関わらず常に「発生しない(False)」となること。

2つのルールの組み合わせ（AND条件）
----------------------
複数のルールが指定された場合、1イベントは「すべてのルールを満たす場合のみ発生する（AND条件）」。
ルールA（元結果T/F、Not T/F） × ルールB（元結果T/F、Not T/F）の全16通りの組み合わせパターンで、最終的なイベント発生有無が論理的に正しいかを確認する。

| ルールA(元) | ルールA(Not) | ルールA(最終) | ルールB(元) | ルールB(Not) | ルールB(最終) | イベント発生(A AND B) |
|---|---|---|---|---|---|---|
| True | False | True | True | False | True | **発生する** |
| True | False | True | True | True | False | 発生しない |
| True | False | True | False | False | False | 発生しない |
| True | False | True | False | True | True | **発生する** |
| True | True | False | (略:Bは4通り) | | | 全て発生しない |
| False | False | False | (略:Bは4通り) | | | 全て発生しない |
| False | True | True | True | False | True | **発生する** |
| False | True | True | True | True | False | 発生しない |
| False | True | True | False | False | False | 発生しない |
| False | True | True | False | True | True | **発生する** |

FromToの観点
-----------------

### EntityFromとEntityTo
- 開始月（FromYM）ちょうどで発生する（境界値）
- 開始月（FromYM）の1ヶ月前で発生しない（境界値）
- 終了月（ToYM）ちょうどで発生する（境界値）
- 終了月（ToYM）の1ヶ月後で発生しない（境界値）

### EventFromとEventTo
- 開始月（FromYM）ちょうどで発生する（境界値）
- 開始月（FromYM）の1ヶ月前で発生しない（境界値）
- 終了月（ToYM）ちょうどで発生する（境界値）
- 終了月（ToYM）の1ヶ月後で発生しない（境界値）

### EntityFrom-EntityToとEventFrom-EventToの重なりパターン
- Entityの期間とEventの期間の**両方を満たす（重なっている）期間のみ有効になる（AND条件）**ことの確認。以下の全パターンにおいて、期待される有効期間（共通期間）が正しく計算されるかを網羅する。
  - **完全一致**: Entity(2024/01-2025/12)、Event(2024/01-2025/12) → 有効期間: 2024/01〜2025/12
  - **包含（Entityが大きい）**: Entity(2024/01-2025/12)、Event(2024/06-2025/06) → 有効期間: 2024/06〜2025/06
  - **包含（Eventが大きい）**: Entity(2024/06-2025/06)、Event(2024/01-2025/12) → 有効期間: 2024/06〜2025/06
  - **前方交差（Eventが先）**: Entity(2024/06-2025/12)、Event(2024/01-2025/06) → 有効期間: 2024/06〜2025/06
  - **後方交差（Entityが先）**: Entity(2024/01-2025/06)、Event(2024/06-2025/12) → 有効期間: 2024/06〜2025/06
  - **重なりなし（Eventが過去）**: Entity(2025/01-2025/12)、Event(2024/01-2024/12) → 有効期間: なし（常にFalse）
  - **重なりなし（Eventが未来）**: Entity(2024/01-2024/12)、Event(2025/01-2025/12) → 有効期間: なし（常にFalse）

利用シーンを想定した観点（結合テスト寄り）
-----------------------

### 車検：2年に1回（例：2024年から2年ごとの5月）
- 想定ルール: `SOME_YEARMONTHS(Start: 2024, Step: 2, Months: [5])`
- 確認: 2024年5月、2026年5月に発生し、2025年5月には発生しない。

### 点検：半年に1回、ただし車検タイミングを除く
- 想定ルール:
  - ルール1: `SOME_MONTHS(Months: [5, 11])`
  - ルール2: `SOME_YEARMONTHS(Start: 2024, Step: 2, Months: [5], Not: true)`
- 確認: 2024年11月、2025年5月、2025年11月には発生する。2024年5月には発生しない。

### 毎月の出費だが、ボーナス月（6月・12月）だけは発生しない出費
- 想定ルール:
  - ルール1: `EVERY_MONTH()`
  - ルール2: `SOME_MONTHS(Months: [6, 12], Not: true)`
- 確認: 5月や7月は発生するが、6月と12月は発生しない。

### 夏季休暇の特別積立（8月）、ただし子が中学生になる2030年まで
- 想定: EventToを `2030-12` に設定 ＋ ルール `SOME_MONTHS(Months: [8])`
- 確認: 2030年8月までは発生するが、2031年8月以降は発生しない。
